## 2026-03-01

### BUG修复

#### 邀请人数统计一致性修复
- 修复 `aff_count` 在并发注册场景下可能少计的问题：由“先查后改再保存”改为数据库原子自增。

- 修复用户页首屏命中“已保存筛选条件”时可能持续转圈的问题。

### 新增功能

#### 邀请体系支持“双机制并存”
- 保留老机制：邀请注册固定奖励（`QuotaForInviter` / `QuotaForInvitee`）。
- 新增新机制：邀请充值返佣（按充值额度比例计算，`T+1` 结算）。
- 两种机制可同时存在、互不冲突。
- 返佣只统计成功充值订单，注册赠送额度不计入返佣。
- 返佣结算时间统一为自然日 `00:00`
- 月卡每日重置时间统一为自然日 `00:00`。
- 修改文件在`constant.go`

### 功能改进

#### OpenAI 渠道重试优化
- 改进前：用户配置多个 OpenAI 渠道时，遇到 `We're currently experiencing high demand`（429），重试可能再次选到刚失败的渠道，切换效果不稳定。
- 改进后：重试选渠道时会带上“已失败渠道 ID 列表”，并从候选中排除这些渠道，避免重复命中失败渠道。
- 改进效果：在多 Key 场景下，429 后的重试更容易切换到可用渠道，提升请求成功率和稳定性。

#### 用户管理与令牌管理优化
- 用户管理筛选增强：支持角色、状态、邀请人 ID、被邀请人 ID、是否有邀请人、是否有被邀请人。
- 支持邀请关系详情：新增接口 `GET /api/user/:id/invite-relations`，并提供侧栏查看当前用户、邀请人、被邀请人（分页）。
- 邀请信息补全：返回 `inviter_username`，避免前端逐行查询造成 N+1 问题。
- 用户/令牌列表均新增 `已使用余额`（`used_quota`）展示与区间筛选。
- 用户管理排序升级：支持 `ID` 与 `额度` 独立排序且可组合使用（如 `ID降序 + 额度升序`）。
- 用户管理筛选布局优化：`额度最小值/最大值` 移入高级筛选，主筛选区更紧凑；搜索提示统一为 `ID/用户名`。
- 令牌管理新增按分组筛选，金额筛选与金额升降序可联动查询，并在翻页/改每页条数/刷新时保持条件。
- 额度筛选语义统一为“总额度”（可用额度 + 已使用额度）；高级筛选分组与文案优化，移除内部字段名外露。
- 新增修复接口 `POST /api/user/rebuild-aff-count`，支持全量重算或按 `user_id` 单用户重算，修复历史 `aff_count` 不一致。

#### 额度稳定性与多 Key 防抖优化
- 新增同渠道多 Key 快速切换：当请求命中“额度类错误”且渠道为多 Key 时，同次请求可在同渠道内立即切换下一个 Key 重试一次，减少“这次失败下次又好”的抖动。
- 新增多 Key 冷却机制：触发额度类错误的 `channel + key_index` 会进入短时冷却（默认 `60s`），冷却期内不再参与选 Key。
- 新增 token 维度的 Key 粘性：同一 `token + channel + model` 在短窗口内优先复用同一上游 Key（默认 `60s`），减少来回切换上游账号导致的波动。
- 新增额度不足前 DB 二次确认开关：仅在即将返回“额度不足”时回源数据库复核，降低缓存短暂不一致导致的误判。
- 优化失败退款一致性：失败链路的额度返还改为优先立即生效，降低“先扣后还”的时间窗口。
- 新增可配置项（可在系统选项中调整）：
  - `QuotaStabilityEnabled`：是否开启额度防抖总开关。默认 `true`。
  - `QuotaInsufficientDBRecheckEnabled`：返回“额度不足”前是否回源 DB 二次确认。默认 `true`。
  - `MultiKeySameChannelFailoverOnce`：额度类错误时，是否在同渠道内立即切换 Key 重试一次。默认 `true`。
  - `MultiKeyCooldownSeconds`：触发额度类错误后，Key 冷却时长（秒）。默认 `60`。
  - `MultiKeyStickySeconds`：同一 `token + channel + model` 的 Key 粘性时长（秒）。默认 `60`。
