# new-api 项目全面优化评估报告

> 评估日期：2026-04-08
> 评估范围：前端性能、后端性能、代码质量、用户体验、安全性

---

## 一、前端性能优化

### 已有良好实践

- 27 个页面全部使用 `React.lazy()` + `Suspense` 路由级懒加载
- Vite `manualChunks` 拆分了 5 组核心依赖（react-core、semi-ui、tools、react-components、i18n）
- 请求竞态保护（`requestCounter`）和最小 Loading 时间 Hook（`useMinimumLoadingTime`）
- 统一 Axios 实例 + 请求去重机制

### 优化项

#### 1. [高优先级] 移除冗余 UI 库 — Semi UI + Antd 并存

- **文件**: `web/package.json`
- **问题**: 同时引入了 `@douyinfe/semi-ui` 和 `antd`，Antd 仅少量使用，但保留两套 UI 库导致包体积增加约 200-300KB (gzip)
- **建议**: 全面迁移到 Semi UI，移除 `antd` 依赖
- **预期效果**: 减少 200-300KB 包体积，加快首屏加载

#### 2. [高优先级] 重型依赖未纳入 `manualChunks`

- **文件**: `web/vite.config.js`
- **问题**: `mermaid`（~800KB）、`@visactor/vchart`（~600KB）、`katex`（~300KB）、`react-markdown` + remark/rehype 链未单独拆分 chunk
- **建议**: 在 `manualChunks` 中增加：
  ```js
  'mermaid': ['mermaid'],
  'vchart': ['@visactor/vchart', '@visactor/react-vchart', '@visactor/vchart-semi-theme'],
  'markdown': ['react-markdown', 'remark-gfm', 'remark-math', 'rehype-highlight', 'rehype-katex'],
  'katex': ['katex'],
  ```
- **预期效果**: 首屏加载提速 30-50%，非必要页面不加载这些重型库

#### 3. [中优先级] `optimizeDeps.force: true` 导致每次启动重新预构建

- **文件**: `web/vite.config.js`
- **问题**: `force: true` 使每次 dev 启动都重新预构建依赖
- **建议**: 移除 `force: true`，仅在依赖变更时触发预构建
- **预期效果**: 开发服务器启动提速 3-5 秒

#### 4. [中优先级] 无 Service Worker / 离线缓存策略

- **问题**: 当前仅使用 `localStorage` 进行缓存（48 个文件使用），无浏览器级缓存策略
- **建议**: 对静态资源考虑添加 Cache-Control headers（需后端配合），或使用 Vite PWA 插件
- **预期效果**: 重复访问加载提速，减少网络请求

#### 5. [低优先级] CSS 方案混合 — Tailwind + Semi UI CSS + PostCSS

- **问题**: 可能存在样式冲突和重复 CSS
- **建议**: 统一 CSS 策略，优先使用 Tailwind utility classes 或 Semi UI token，避免混用
- **预期效果**: 减少 CSS 体积，避免样式冲突

## 二、后端性能优化

### 已有良好实践

- 三级缓存架构（内存 → Redis → DB），热路径几乎零 DB 查询
- 连接池配置合理（MaxIdle=100, MaxOpen=1000）
- 启用 `PrepareStmt` 预编译提升重复查询性能
- 批量更新聚合机制减少数据库写入
- Channel 内存缓存 + 加权随机选择，路由决策零 DB 查询
- 多层次限流体系（IP/用户/令牌/模型/系统）

### 优化项

#### 1. [高优先级] N+1 查询 — `BatchSetChannelTag`

- **文件**: `model/channel.go:970-1001`
- **问题**: 批量设置 N 个渠道 tag 时，产生 N 组 DELETE + INSERT 操作。每个 `UpdateAbilities` 先删除该渠道所有 Ability，再重新插入
- **建议**: 合并为 2 条 SQL：
  ```go
  tx.Where("channel_id IN ?", ids).Delete(&Ability{})
  tx.CreateInBatches(allAbilities, 100)
  ```
- **预期效果**: 批量操作提速 10-100x

#### 2. [高优先级] 逐条 UPDATE — `UpdateChannelClientRestrictionByTag`

- **文件**: `model/channel.go:768-797`
- **问题**: 同一 tag 下的渠道逐条 UPDATE，未使用批量更新
- **建议**: 改为单条 `UPDATE channels SET setting = ? WHERE tag = ?`
- **预期效果**: 减少数据库往返次数

#### 3. [高优先级] 批量更新器逐条刷盘

- **文件**: `model/utils.go:76-94`
- **问题**: 不同实体仍逐条执行 SQL，虽然已合并同一实体的多次更新
- **建议**: 改用批量 `INSERT ... ON DUPLICATE KEY UPDATE` 或 `CASE WHEN` 模式
- **预期效果**: 减少大量活跃用户场景下的数据库压力

#### 4. [高优先级] Channel `status` 字段缺少索引

- **文件**: `model/channel.go`
- **问题**: 多处代码按 `status` 查询（`DeleteDisabledChannel`、`GetChannelsByTag` 等），但 `status` 字段无索引
- **建议**: 添加索引 `ALTER TABLE channels ADD INDEX idx_channel_status (status);`
- **预期效果**: 状态相关查询走索引，避免全表扫描

#### 5. [中优先级] Group/Models 字段逗号分隔存储无法走索引

- **文件**: `model/channel.go:330-335`
- **问题**: `Group` 和 `Models` 字段存储为逗号分隔字符串（如 `"group1,group2"`），查询使用 `LIKE '%,group,%'` 前缀通配符导致全表扫描
- **建议**: 考虑使用关联表（`channel_groups`）替代逗号分隔存储，或 PostgreSQL 使用数组类型 + GIN 索引
- **预期效果**: 渠道搜索性能大幅提升（从 O(n) 到 O(log n)）

#### 6. [中优先级] Redis 限流器非 Pipeline 操作

- **文件**: `middleware/rate-limit.go:21-66`
- **问题**: 每次请求 2-3 次 Redis 命令（LPUSH + LINDEX + LTRIM），非 Pipeline
- **建议**: 改用 Redis Lua 脚本，将多次操作合并为一次调用
- **预期效果**: 高并发下减少 Redis RTT，降低延迟

#### 7. [中优先级] 内存限流器全局锁

- **文件**: `common/rate-limit.go:86-111`
- **问题**: `InMemoryRateLimiter` 使用单一 `sync.Mutex`，所有限流 key 共享同一把锁
- **建议**: 改用分片锁（`sync.Map` + 分段锁）减少锁竞争
- **预期效果**: 高并发下减少锁竞争，提升吞吐量

#### 8. [低优先级] Token 认证每次请求反序列化完整对象

- **文件**: `middleware/auth.go`
- **问题**: `TokenAuth` 中间件每次请求从 Redis Hash 反序列化整个 Token 对象，Token 模型字段较多
- **建议**: 考虑只缓存 Token 的核心字段（key、status、quota、expired_time）到精简结构
- **预期效果**: 减少 Redis 数据传输量和反序列化开销

## 三、代码质量提升

### 已有良好实践

- 页面/组件/Hook 三层分离，页面入口极简（~30行）
- 统一 API 响应约定 `{ success, message, data }`
- LIKE 搜索模式清洗（`sanitizeLikePattern`）
- 排序字段白名单校验防止 ORDER BY 注入
- 跨数据库兼容性处理（`commonGroupCol`、`commonTrueVal` 等）

### 优化项

#### 1. [高优先级] `helpers/utils.jsx` 职责过重（952 行）

- **文件**: `web/src/helpers/utils.jsx`
- **问题**: 混合了 Toast 通知、时间格式化、价格计算、分页工厂、Playground 消息处理等不相关功能
- **建议**: 拆分为独立模块：
  - `helpers/toast.js` — Toast 通知（showError/showSuccess/showWarning/showInfo/showNotice）
  - `helpers/price.js` — 价格计算与格式化
  - `helpers/date.js` — 时间格式化工具
  - `helpers/pagination.js` — 分页工厂函数
  - `helpers/playground.js` — Playground 消息处理
- **预期效果**: 提高可维护性，便于单元测试，减少合并冲突

#### 3. [高优先级] Hook 内大量重复代码

- **文件**: `web/src/hooks/users/useUsersData.jsx`（722行）、`web/src/hooks/tokens/useTokensData.jsx`（602行）
- **问题**: 两个 Hook 存在大量重复的参数校验（`!== '' && !== null && !== undefined`）、分页、搜索、loading 控制模式
- **建议**: 创建通用 `usePaginatedData` 工厂 Hook：
  ```jsx
  function usePaginatedData({ fetchFn, searchParams }) {
    const [loading, setLoading] = useState(true);
    const [searching, setSearching] = useState(false);
    const [data, setData] = useState([]);
    const [page, setPage] = useState(1);
    // 统一分页、搜索、竞态保护、loading 控制逻辑
  }
  ```
- **预期效果**: 减少 50%+ 重复代码，统一数据加载模式

#### 4. [中优先级] Tokens 组件复杂度过高（496行）

- **文件**: `web/src/components/table/tokens/index.jsx`
- **问题**: 混合了 FluentRead 浏览器扩展集成、MutationObserver DOM 监听、模型加载等不相关职责
- **建议**: 将 FluentRead 集成抽取为独立的 `useFluentIntegration` Hook，DOM 监听抽取为 `useMutationObserver` 通用 Hook
- **预期效果**: 组件职责清晰，便于独立测试和维护

#### 5. [中优先级] `showError` 中硬编码中文未走 i18n

- **文件**: `web/src/helpers/utils.jsx:134-143`
- **问题**: 405/500 等错误提示使用硬编码中文字符串，未使用 `t()` 函数
- **建议**: 将错误提示文案提取到 i18n 资源文件中，使用 `t('错误提示key')` 替代硬编码
- **预期效果**: 完善国际化支持

## 四、用户体验改进

### 已有良好实践

- 完善的 5 级 Toast 通知体系（error/success/warning/info/notice）
- `showError()` 对 401（自动跳转登录）、429（限流提示）、500（服务器错误）等状态码差异化处理
- 最小 Loading 时间 Hook 防止骨架屏闪烁
- 响应式设计（`useIsMobile` 使用 `useSyncExternalStore` 并发安全 API）
- 侧边栏状态持久化
- 主题切换（亮色/暗色/自动）+ 系统主题偏好监听

### 优化项

#### 1. [中优先级] 缺少全局 API 请求 Loading 指示器

- **问题**: 当前依赖各组件自行管理 loading 状态，页面切换时无统一的视觉反馈
- **建议**: 在 Axios 拦截器中添加全局进度条（如 NProgress）：
  ```js
  API.interceptors.request.use((config) => {
    NProgress.start();
    return config;
  });
  API.interceptors.response.use((response) => {
    NProgress.done();
    return response;
  });
  ```
- **预期效果**: 页面切换和数据加载时有统一的视觉反馈

#### 2. [中优先级] ProfitBoard 缓存版本号机制

- **文件**: `web/src/pages/ProfitBoard/hooks/useProfitBoardPersist.js`
- **问题**: 已有 localStorage 缓存报表数据，但缺少缓存版本号机制，数据结构变更时可能出现兼容性问题
- **建议**: 添加缓存版本号，版本不匹配时自动清除旧缓存
- **预期效果**: 避免缓存数据结构不兼容导致的页面异常

#### 3. [低优先级] 表格批量操作进度反馈

- **问题**: 批量操作（批量删除、批量启用/禁用）时缺少进度指示
- **建议**: 添加操作进度条或进度计数器（如 "正在处理 3/10..."）
- **预期效果**: 提升批量操作时的用户信心

## 五、安全性增强

### 已有良好实践

- **密码处理优秀**: bcrypt 哈希（cost=10）、恒定时间比较、密码字段 `Omit` 排除
- **SQL 注入防护优秀**: 全面使用 GORM 参数化查询、排序白名单、LIKE 清洗
- **SSRF 防护完善**: 私有 IP 检测、域名/IP 白名单、端口限制、DNS 重绑定防护
- **多层次限流**: 全局/IP/用户/令牌/模型/系统 6 级限流，Redis + 内存双后端
- **认证体系完善**: 三级角色体系（Root/Admin/Common）、双重认证（Session + AccessToken）、IP 白名单
- **安全验证机制**: 敏感操作需要 5 分钟内通过验证
- **WebAuthn/Passkey**: 支持 FIDO2 无密码登录
- **Cloudflare Turnstile**: 注册人机验证

### 优化项

#### 1. [高优先级] CORS 配置过于宽松 — CSRF 风险

- **文件**: `middleware/cors.go:9-16`
- **问题**: `AllowAllOrigins = true` + `AllowCredentials = true`，允许任意域发起携带 Cookie 的请求，等效于信任所有来源
- **建议**: 生产环境限制为已知前端域名：
  ```go
  config.AllowOrigins = []string{"https://your-domain.com", "https://admin.your-domain.com"}
  config.AllowCredentials = true
  ```
- **预期效果**: 修复 CSRF 漏洞，防止跨站请求伪造攻击

#### 2. [高优先级] TLS 证书验证跳过 — 中间人攻击风险

- **文件**: `common/constants.go:77-78`
- **问题**: `InsecureSkipVerify: true` 跳过上游服务器证书验证，可能遭受中间人攻击（MITM）
- **建议**: 确保生产环境 `TLSInsecureSkipVerify` 设为 `false`，仅开发/测试环境使用
- **预期效果**: 防止中间人攻击，确保与上游 API 通信安全

#### 3. [中优先级] Access Token 明文存储

- **文件**: `model/user.go:1375-1385`
- **问题**: `access_token` 字段以 `char(32)` 明文存储在数据库中，数据库泄露将直接暴露所有用户的 Access Token
- **建议**: 存储 Access Token 的 SHA256 哈希值，验证时对传入 token 做哈希后比对
  ```go
  func ValidateAccessToken(token string) *User {
      hash := sha256.Sum256([]byte(token))
      hashStr := hex.EncodeToString(hash[:])
      // 用 hashStr 查询数据库
  }
  ```
- **预期效果**: 即使数据库泄露，攻击者也无法获取可用的 Access Token

#### 4. [中优先级] 渠道 API Key 明文存储

- **文件**: `model/channel.go`
- **问题**: 所有上游提供商的 API Key 以明文存储在数据库中，数据库泄露将导致所有渠道密钥暴露
- **建议**: 使用 AES-256-GCM 加密存储，运行时解密。可复用 `common/crypto.go` 中的 `CryptoSecret` 作为主密钥
- **预期效果**: 数据库泄露不暴露渠道密钥

#### 5. [中优先级] Session 密钥管理

- **文件**: `common/constants.go`
- **问题**: `SessionSecret` 使用 UUID 随机生成，应用重启后所有现有 Session 失效
- **建议**: 通过环境变量 `SESSION_SECRET` 设置固定密钥，支持多实例部署时共享 Session
- **预期效果**: 避免重启导致用户掉线，支持多实例部署

#### 6. [低优先级] `TokenAuthReadOnly()` 宽松模式审计

- **文件**: `middleware/auth.go`
- **问题**: 即使令牌已过期/耗尽/禁用也允许访问，虽然注释说明仅用于只读查询接口
- **建议**: 确保仅在真正的只读查询端点使用，且添加审计日志记录
- **预期效果**: 防止误用于写操作端点
